Задание № 1 - разбивка монолитного фронтэнда на микрофронтэнды

ВНИМАНИЕ! Я никогда ранее не писал на JS, никогда ранее не был разработчиком фронтэнда, и про React только слышал краем уха. Я потратил чудовищное количество времени на это задание, но, несмотря на это, вероятно, в моем решении будут заметны очень странные для опытного фронтэндера вещи. Тем не менее, каким-то непостижимым образом разбитый мной на микрофронты сервис работает, и я реализовал все три уровня решения первого задания.

Я выбрал Module Federation для разбивки, т.к. сервис написан с использованием одного-единственного фреймворка  React, и потребностей в других фреймворках на данный момент нет и не предвидится. Singla SPA был бы более уместен, если бы разные части сервиса использовали бы разные фреймворки, или если бы в этом сервисе дизайн подразумевал несколько сложных и достаточно независимых друг от друга микрофронтов, возможно, отображаемых одновременно. Здесь же у нас по факту бОльшая часть сервиса крутится вокруг доски карточек, остальное всё простенькое, и всплывает на экране достаточно редко. Таким образом, главные преимущества Single SPA для сервиса Mesto не совсем актуальны. Зато в Module Federation очень легко разделять компоненты между микрофронтами (например, PopupWithForm).

Запуск всех микрофронтов: docker-compose up -d в корневой папке. Если запускаете Docker в винде или на маке, не забудьте сначала включить в настройках Docker Desktop флаг Enable host networking. Когда всё запустится, открывайте браузер на странице http://localhost:8095

Теперь к решению. Монолит был разбит на следующие микрофронты:

1. **auth** - микрофронт регистрации и авторизации (компоненты CheckToken, InfoTooltip, Login и Register). CheckToken по факту не компонент, а просто функция, проверяющая валидность токена. InfoTooltip - сообщение о успешной или неуспешной регистрации/авторизации. Login и Register - компоненты форм авторизации и регистрации. Успешные или неуспешные регистрация и авторизация пользователя приводят к возникновению событий reg-ok, reg-fail, login-ok и login-fail. Файл utils/auth.js содержит только нужные в этом микрофронте функции. В папке blocks лежат только нужные для этого микрофронта стили.
Данный микрофронт никак не зависит от остальных микрофронтов. Сервис, обслуживающий этот микрофронт, запускается на порту 8097.

2. **cardboard** - микрофронт доски с карточками-изображениями. Содержит компоненты Card (карточка с изображением, кнопкой like и удаления), ImagePopup (всплывшее по щелчку мыши увеличенное изображение), Main (собственно, доска из карточек), ProtectedRoute (обертка для компонентов в маршрут-Route). Все эти компоненты, кроме Card, экспортируются. Микрофронт запускается на порту 8096, и зависит от микрофронтов profile и upload (см. ниже). Также микрофронт предоставляет (является провайдером контекста) разделяемую между несколькими микрофронтами библиотеку usercontext (я не нашел более изящного решения для протаскивания контекста, хранящего информацию о текущем пользователе, между разными микрофронтами).

3. **host** - основной (host) микрофронт. Содержит главное приложение (App), заголовок и низ страницы (Header и Footer) и компонент PopupWithForm (всплывающее окно, используется также и в других микрофронтах). Обеспечивает маршрутизацию. Зависит от cardboard и auth. Запускается на порту 8095.

4. **profile** - микрофронт профиля пользователя. Содержит компоненты EditAvatarPopup (редактирование аватара пользователя), EditProfilePopup (редактирование профиля пользователя) и заголовок доски карточек с информацией о пользователе, кнопками редактирования профиля и аватары, кнопки вызова загрузки новой карточки. Зависит только от микрофронта host, откуда берет PopupWithForm (вероятно, за этим компонентом лучше было бы ходить не в host, а в разделяемую библиотеку, но мне лень для единственного компонента эту библиотеку заводить). Запускается на порту 8098.

5. **upload** - микрофронт загрузки новой карточки (изображения). Выделен в отдельный микрофронт, поскольку в перспективе может обрасти сложным функционалом по групповой загрузке, импорту из разных хранилищ, и т.д. Зависит от микрофронта host, откуда берет компонент PopupWithForm, содержит только компонент AddPlacePopup. Микрофронт запускается на порту 8099.

В процессе разбивки я также урезал обширный файл api.js из папки utils монолита для каждого микрофронта в соответствии с тем функционалом, что микрофронт предоставляет. Микрофронт auth вообще содержит только auth.js, остальные содержат урезанные api.js
